Critical Money

Project documentation draft brief summary to guide how the contract works:

First function (Contract: Attorneycoin):
function mintToUser() external;

  • This function allows any investor to go to the faucet and mint payment cryptocurrencies created specifically to drive the platforms pawn economy. It has no parameters, no modifiers; anyone who connects their wallet can call the function.

Second function (Contract: LendingPool):
function lendUSDC(uint256 amount) external nonReentrant;

  • This function enables investors to make decentralized deposits into the lending pool, increasing the platforms reserve and providing resources for loans. The parameter is the amount of payment tokens to deposit. Its similar to a transfer function, but it also records the investors address and adds the principal amount to a UserData struct for better management.
    Note: The withdraw function follows the same logic, but instead of adding, it subtracts.

Third function (Contract: PledgePlatform):
function pledgeRegistry(address _pledgor, uint256 amount, uint256 spread) external onlyPledgee;

  • This function is used to formalize the pledge. Input parameters are the address of the borrower (pledgee), the amount of metal ounce tokens, and the spread percentage. This function is more complex, so heres the logic: it authorizes the whitelist, fetches the metal price into the contracts memory, records the pledge data in the Pledge struct, adds the pledge ID to the idByPledgor mapping (so you can later retrieve the list of IDs an investor has by address), updates the agreementId number, mints the metal tokens to the investors address, calculates the loan amount to be granted, and calls the borrow function to initiate the loan.

The borrow function calculates the spread by assigning the number to an equivalent percentage, deducts the spread and returns the net value via the calculateCreditWithSpread function, then subtracts the net value from the loan amount, returning the total spread amount. Then, theres an if-else table that calculates the reward rate percentage, separates the total reward from the platform fee, transfers the net value to the borrower, calls the function that calculates how much will be distributed to each investor according to their deposit percentage in the lending pool reserves via the distributeRewards function (which saves in a rewards mapping how much each investor is entitled to claim), and finally transfers the platform fee to the creditor/administrators address, updating the total concentrated pool reserve.

Fourth function (Contract: PledgePlatform):
function amortizePledge(uint256 amount, uint256 pledgeId) external virtual returns (uint256);

  • This function is used to pay off the debt and redeem the pledge later. You need to provide the amount you want to pay at that moment and the pledge ID to identify the debtor and proceed with the records and updates in their Pledge struct. Payments can be made in installments, but the full amount must be amortized by the scheduled redemption date in the Pledge struct. There is validation that reverts if payment above the amount due is attempted. First, it checks how much has already been paid via the paymentsMade mapping, then deducts the debt amount from the amount saved in the mapping. The amount is then transferred to the LendingPool contract, which is where the total reserve and each investors principal are recorded. Then, it executes the reserve update function called updateReserve, adds the paid amount to the payments mapping, and finally checks if the redemption has already been made via a boolean true or false. If true, the function reverts.

Fifth function (Contract: PledgePlatform):
function executeWarranty(uint256 pledgeId) external onlyPledgee returns (bool);

  • This function allows the administrator to execute the guarantee centrally if the redemption period registered in the pledge struct has already elapsed, allowing them to choose to execute the guarantee or wait longer for payment, enabling speculation with the custodied metal. The pledge ID must be provided as a parameter for identification and handling. It checks if the ID exists and is associated with a debtor, then checks if the redemption has already been approved (if so, the function reverts via the boolean true), then checks if the redemption date has passed, and if all checks pass, returns true for the redemption.

Note: Obviously, there are several other functions that can be created and will be created later. However, at this moment, these are the most basic functions for the platforms minimum operation, and it is already possible to build a functional frontend.
